Commit Message Format Guidelines

A well-structured commit message enhances code readability and project maintenance. Follow the guidelines below to ensure consistency and clarity in your commit messages.

STRUCTURE

Each commit message should adhere to the following format:

<type>(<scope>): <subject>
1. TYPE

Description: The type indicates the purpose of the commit.

Options: Use one of the following predefined types:

  • feat: A new feature.
  • fix: A bug fix.
  • change: A broad category for non-breaking changes.
  • chore: Routine tasks like refactoring, dependency updates.
  • docs: Documentation changes.
  • style: Code style changes (formatting, missing semicolons, etc.).
  • perf: Performance improvements.
  • test: Adding or updating tests.

Case: Lowercase (e.g., feat, fix, change).

2. SCOPE

Description: The scope provides context about the area of the codebase the commit affects.

Format: Use a short, descriptive noun in kebab-case (e.g., module, component, file-name).

Examples:

  • fix(auth-service): correct token validation
  • feat(ui-button): add loading spinner
3. SUBJECT

Description: A concise description of the change.

Format: Write in the imperative mood, as if giving a command or instruction.

Case: Lowercase (e.g., add, fix, update).

Avoid:

  • Ending with a period (.).
  • Starting with a capital letter unless the subject is a proper noun.
4. LENGTH

Subject line: The subject line should not exceed 100 characters.

Body (optional): If necessary, add a more detailed description in the body after a blank line following the subject.

5. CONTAINS A VERB

Description: The commit message must contain at least one verb.

Example:

  • feat(ui): add loading spinner
  • docs(readme): update installation steps
6. VERB TENSE

Description: The script checks if the verb in the commit message is in the past tense. The recommended practice is to use the imperative mood (present tense) for commit messages.

Wrong Example:

  • docs(readme): updated installation steps

Correct Example:

  • docs(readme): update installation steps

Example Commit Messages

Good Examples:
  • chore(set-up): add ESLint rules for code linting
  • docs(file): add README file with project overview
  • fix(auth-service): correct token validation
  • chore(deps): update dependency versions
  • docs(readme): clarify installation steps
  • test(unit): add test cases for login component
  • feat(ui): lorem ipsum dolor sit amet consectetur, adipisicing elit. Nostrum, dolorum.

  • chore(Set-up): add ESLint rules for code linting
  • docs(file): Add README file with project overview

  • docs(readme): clarify installation steps.
  • test(unit): added test cases for login component

    max char length: 100

Adhering to these guidelines will ensure that your commit history remains clean, informative, and easy to navigate.
